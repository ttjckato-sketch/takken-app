# MemoryRecall P38 Input Quality Layer Plan

## 1. 背景
これまでのフェーズ（P1〜P37）では、科学的復習（FSRS）の実装、弱点の特定、特定分野の特訓、および改善スコアの可視化といった「アウトプット中心」の学習サイクルを確立してきた。
しかし、単なる暗記だけでは応用力や制度の深い理解が不足し、初見の「ひっかけ問題」や「複合問題」への対応に限界が生じる。
P38 では、アウトプットの質を支える「インプット教材層（Input Quality Layer）」を再定義・設計し、**理解（Input）→ 定着（Output）→ 改善（Analytics）** がシームレスに繋がる学習OSへと進化させる。

## 2. 質の良いインプットの定義
本システムにおける「質の良いインプット」とは、単なる解説テキストではなく、以下の要素を備えた **「構造化された知識単元（Input Unit）」** を指す。

- **結論ファースト**: 一言でその制度の本質がわかる。
- **目的・趣旨**: なぜこの制度が存在し、試験でどう問われるかがわかる。
- **構成の明確化**: 要件（条件）と効果（結果）が分離されている。
- **原則と例外**: 基本ルールと、試験で狙われる例外パターンが峻別されている。
- **具体例と反例**: 「できる場合」と「できない場合」のイメージを固定する。
- **横断比較**: 似た制度（例：制限行為能力者と意思無能力、35条と37条）との決定的な違いを示す。
- **罠の予告**: 過去の出題傾向に基づく「ひっかけポイント」を明示する。
- **1問確認**: 理解度を即座に判定する最小単位のアウトプットを内包する。
- **補修導線**: 誤答時に「どのインプット単元に戻ればよいか」を自動提示する。

## 3. Input Unit Template の設計
物理ファイルや DB に保存する際の標準フォーマットを定義する。

| フィールド名 | 説明 | 必須 |
| :--- | :--- | :---: |
| `unit_id` | 一意識別子 | ◯ |
| `title` | 制度・論点名 | ◯ |
| `conclusion` | 一言結論（1行） | ◯ |
| `purpose` | 制度の趣旨・試験での位置づけ | ◯ |
| `requirements` | 成立要件（箇条書き） | ◯ |
| `legal_effect` | 法的効果（箇条書き） | ◯ |
| `principle` | 原則ルール | ◯ |
| `exceptions` | 例外ルール（「ただし〜」） | ◯ |
| `cases` | 具体例（Case A: ◯, Case B: ×） | △ |
| `comparison` | 比較対象タグと相違点 | △ |
| `trap_points` | 出題される罠のパターン | ◯ |
| `memory_hook` | 覚え方・ゴロ・図解のヒント | △ |
| `check_question` | 確認用の1問（ActiveRecall形式） | ◯ |
| `linked_tags` | 関連する weak_tags | ◯ |
| `source_trace` | 根拠条文・過去問 ID | ◯ |

## 4. インプットとアウトプットの接続設計
### 理解から定着へのフロー
1.  **Input View**: 上記テンプレートに基づき構造化表示。
2.  **Immediate Check**: 学習直後に `check_question` を解く。
3.  **SRS Registration**: 正解後、FSRS（ActiveRecall/MemoryRecall）の対象として自動登録。
4.  **Looping**: 復習期限が来たら、再びアウトプットを行い、必要に応じてインプット（解説）へ戻る。

### 誤答補修（Repair Loop）の設計
問題（Active/Memory/Trap 等）を間違えた際、以下の順序で情報を提示する。
1.  **なぜ間違えたか（Mistake Analysis）**: 「ひっかけにハマった」「要件を忘れた」等の分類。
2.  **補修説明（Repair Note）**: 当該論点の Input Unit の要点（図解・比較表）をダイジェスト表示。
3.  **関連特訓**: 同一タグの Focus Mode (10Q) への導線提示。

## 5. 技術的整合性と安全制約
- **Schema v24 維持**: P38/P39 では DB 移行を行わず、`metadata` へのキャッシュまたは `understanding_cards` の動的拡張によりプロトタイプを構築する。
- **情報の分離**:
    - **Dashboard**: 「次に読むべきインプット」を推奨アクションとして表示。
    - **db-audit**: インプット品質（網羅率、required field 充足率）を監査。
- **回帰ガード**: 既存の 30問/35問セッション、`auto_apply` の安定動作には一切干渉しない。

## 6. 今後のステップ（P39〜P43）
- **P39**: Input Unit Template の最小実装とダミーデータ表示。
- **P40**: 宅建業法（重要論点）の構造化インプット生成。
- **P41**: 誤答画面からの「補修インプット」ポップアップ実装。
- **P42**: 制度比較表のインプット層統合（ComparisonRecall の強化）。
- **P43**: 全分野（権利関係・制限・賃管）への拡大展開。

## 7. Acceptance Criteria (AC)
- **AC-001**: 質の良いインプットの定義が明文化されている。
- **AC-002**: Input Unit Template の全フィールドが定義されている。
- **AC-003**: 誤答補修（Repair Loop）の表示フローが設計されている。
- **AC-004**: v24 Schema を変更せずに実装可能であることが確認されている。
- **AC-005**: Build PASS / 既存機能（Focus Mode, weak_score_delta）の安定維持。
