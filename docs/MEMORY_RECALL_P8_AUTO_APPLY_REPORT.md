# MemoryRecall P8 Auto Apply Report

## 概要
FSRS 指標に基づいたセッション配分の「完全自動適用 (Auto Apply)」機能を実装した。
P7 の手動予約フローを土台としつつ、厳格な安全条件（正答率 75% 以上、未完了ゼロ等）を自動判定し、インテリジェントに学習バランスを最適化する仕組みを構築した。

## 実施内容
- **自動適用資格（Eligibility）エンジンの実装**: `evaluateAutoApplyEligibility` を追加。直近7日の実績から「自動変更してよいか」を多角的に評価。
- **動的実行レイヤーの統合**: `buildDailyStudySessionQueue` において、手動予約（P7）を最優先しつつ、予約がない場合に資格を満たせば自動適用を行うロジックを実装。
- **一回使い切り・クールダウン管理**: 自動適用後は 20時間のクールダウンを設定し、1日最大1回の更新に制限。
- **Rollback の強化**: ロールバック実行時に `auto_apply_enabled` を自動で OFF にし、24時間の再有効化禁止期間を設定。
- **監査 UI の拡張**: `db-audit.html` に自動適用のトグルスイッチ、ブロック理由の詳細表示、および全リセットボタンを追加。

## 結果 (実測値)
- **初期状態**: `auto_apply: OFF` (規定通り)
- **自動適用範囲**: Active 22-26 : Memory 4-8 (合計 30固定)
- **監査ログ**: `applied_by: "auto"` / `"manual"` の識別子を `metadata` に記録。
- **Build 結果**: PASS (TypeScript / Vite)
- **Schema 整合性**: FULL_SYNC (v24) 維持

## 動作確認
- 初期状態（OFF）で配分 24:6 が維持されることを確認。
- 手動予約（P7）がある場合に自動適用が正しくスキップされ、予約が優先されることを物理 Trace 監査済み。
- ロールバックボタンにより、設定が 24:6 へ回帰し、自動適用が即座に無効化されることを確認。
- `db-audit.html` において、不適格な場合に具体的なブロック理由（「Active正答率不足」等）が表示されることを確認。

## P9 (次フェーズ) への課題
- **長期最適化**: 1ヶ月以上の長期トレンドに基づいた、より深い FSRS パラメータの自動チューニング。
- **多モード対応**: NumberRecall / TrapRecall 等の特化型カードを動的スロットへ統合。
- **App 内設定 UI**: 開発者用 HTML ではなく、一般ユーザー向け設定画面への移植。
