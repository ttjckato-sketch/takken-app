## 2. 承認基準 (Review Criteria)
`human_review_required` から `auto_ok` (または直接昇格) へ移行するための必須条件。

### 承認可 (Approve) の条件:
1. **テキストの自然さ**: `restored_text` が問題文として成立していること。
2. **正誤判定の確定**: 外部ソース（RETIO 正解表、法令条文）に基づき、`restored_is_true` が物理的に裏付けられていること。
3. **法的根拠の明示**: `restored_explanation` に条文番号や判例、公的統計への言及が含まれていること。
4. **ソース参照の完備**: `source_refs` に有効な URL または文献名が登録されていること。
5. **法改正対応**: 旧問の場合、現行法（例：2020年民法改正、2021年賃貸住宅管理業法）に照らして正誤が有効であることを確認済みであること。

### 却下 (Reject) の条件:
1. **復元不能**: 文脈から問題文や正誤判定を推測できない場合。
2. **法的矛盾**: 解説文と正誤判定の間に論理的な乖離がある場合。
3. **低品質**: プレースホルダーが解消されず、学習資産としての価値が低い場合。

## 3. 個数問題・組み合わせ問題の扱い
個数問題の各肢は、単体での〇×判定が不可能なケース（例：「適切なものはいくつあるか」に対する「1つ」という肢）が多いため、以下のいずれかの対応をとる。
- **原則**: `review_status: rejected` (肢単位の学習には不適)。
- **例外**: 解説文から「アは適切、イは不適切」といった個別判定が 100% 抽出可能な場合に限り、問題文を再構成して `auto_ok` へ昇格。

## 4. 承認・昇格プロセス (Step-by-Step)
1. **目視確認**: `db-audit.html` または JSON データ上で候補を精査。
2. **パッチ適用**: 根拠に基づき `restoration_candidates` の `restored_` フィールドを更新し、`review_status` を `auto_ok` に変更。
3. **昇格 dry-run**: 昇格実行前に ID 衝突や品質をシミュレーション。
4. **物理バックアップ**: 昇格直前に IndexedDB 全体をエクスポート。
5. **本昇格実行**: 正式な `knowledge_units` / `memory_cards` テーブルへマージ。

## 5. 安全上の制約
- **Source Choices 保護**: 元データ（`source_choices`）は絶対に直接上書きしない。
- **履歴分離**: 復元されたカードの学習履歴は `memory_study_events` に隔離し、ActiveRecall の統計を汚染しない。
- **一括昇格禁止**: 2621 件の全件復元は、本ワークフローによる PoC 完了まで実行禁止。
