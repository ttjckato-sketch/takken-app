# MemoryRecall P27 Recency Decay Implementation Report

## 概要
学習履歴（weak_tags）の分析精度を向上させるため、**Recency Decay（新鮮度減衰）** エンジンを実装した。
これにより、過去の古いミスよりも直近の躓きを重く反映し、現在の実力に即した「苦手 Top 5」の提示が可能になった。

## 実施内容
- **Index-based Decay の実装**:
  - `analytics.ts` に `calculateDecayFactor` を追加。直近のイベントから順にインデックスを付与し、1.0（最新）〜0.2（最古）の係数を算出。
  - 直近 10件については 0.9以上の高重みを維持する保護ロジックを搭載。
- **weak_tags 算出エンジンの拡張**:
  - `calculateWeakTagsMetrics` を更新し、`final_score = base_score * decay_factor` を適用。
  - db-audit 用に「減衰後スコア」と「生スコア」の両方を保存するように拡張。
- **監査 UI の強化**:
  - `db-audit.html` を P27 仕様に更新。減衰戦略（Index-based）の明示と、スコアリングの前後比較表示を追加。
- **Dashboard の軽量アップデート**:
  - 推奨アクション欄の注釈を更新し、「直近の傾向を重視」していることを学習者に通知。

## 結果 (実測値)
- **減衰範囲**: 1.0 (Index 0) 〜 0.2 (Index 100)
- **Top 5 精度**: 向上（直近 1週間で頻発している分野が上位に来ることを物理監査で確認）
- **Build 結果**: PASS
- **Schema 整合性**: FULL_SYNC (v24) 維持

## 動作確認
- `db-audit.html` において、Index に応じた減衰係数が正確に算出・適用され、苦手 Top 5 が最新トレンドに同期していることを確認。
- 30問標準・35問集中の各セッション、および FSRS 更新フローが、減衰ロジック導入後も正常に機能することを回帰確認。

## P28 (次フェーズ) への課題
