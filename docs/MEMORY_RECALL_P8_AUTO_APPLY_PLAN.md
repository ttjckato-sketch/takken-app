# MemoryRecall P8 Auto Apply Plan

## 1. 背景
MemoryRecall P7 において、推奨配分の永続保存と手動適用予約（1回使い切り）の仕組みを確立し、実測値に基づいた安全な配分調整が可能になった。
P8 では、一定の安全条件を満たした場合にのみ、手動操作なしで配分を自動調整する **完全自動適用 (Auto Apply)** を導入する。

## 2. 目的
- ユーザーの学習パフォーマンスが高い場合に、自動的に最適な学習バランス（Active/Memory 比率）へ誘導する。
- 低正答率や未完了セッションなどのリスクを自動検知し、安全にデフォルト配分（24:6）へ維持・回帰させる。
- 自動調整の履歴を正確に記録し、透明性の高い監査ログを提供する。

## 3. 自動適用を ON にしてよい条件 (Enable Conditions)
以下の条件を**すべて**満たす場合のみ、自動適用を有効化できる。
- **継続性**: 直近 7日間で `incomplete_session_days` が 0。
- **高演習精度**: `active_accuracy_7d` が 75% 以上。
- **低負荷**: `rating_1_2_ratio`（まだ/ギリギリ）が 30% 以下。
- **推奨の安定性**: 推奨配分が 2回連続で同じ方向（例: ともに Active 22: Memory 8）を示している。
- **エラーフリー**: 直近 24時間の Console / Version Error が 0。

## 4. 自動適用の禁止・停止条件 (Blocking Conditions)
以下のいずれか 1つでも満たす場合、自動適用は即座に停止（デフォルト回帰）される。
- **精度低下**: `active_accuracy_7d` が 70% 未満。
- **挫折の兆候**: 未完了セッション（`incomplete_session_days` > 0）が発生。
- **消化不良**: `rating_1_2_ratio` が 40% を超過。
- **手動介入**: ユーザーが手動でロールバック、または手動配分予約を行った。
- **システム異常**: Schema ミスマッチやデータ読み取りエラーの発生。

## 5. 自動適用の頻度と変更幅
- **頻度**: 最大 1日 1回まで。
- **変更幅**: 1回の自動適用による配分変更は最大 ±2問（例: 24:6 -> 22:8）。
- **確認期間**: 自動適用後は最低 1セッションの結果が出るまで次の変更をロックする。

## 6. 監査ログ設計
- **`metadata`**: `auto_apply_active: true` フラグと最終自動適用日時を記録。
- **`study_sessions`**: `applied_by: "auto_apply_p8"` のタグと共に、適用時の算出指標を同梱保存。
- **`study_events`**: 適用された配分下での各問題の結果を従来通り記録。

## 7. db-audit UI 設計
- **ステータス表示**: 現在の `auto_apply` ON/OFF 状態の表示。
- **トグルボタン**: 自動適用の有効化/無効化ボタン（ON条件未達時は無効表示）。
- **停止理由**: 自動適用がブロックされている場合の具体的な理由（例: 正答率不足など）を表示。
- **Rollback**: 自動適用された配分を強制的に 24:6 に戻す緊急ボタン。

## 8. Acceptance Criteria (AC)
- **AC-001**: 条件を満たさない限り `auto_apply` を有効化できない。
- **AC-002**: 有効化中、安全制約（Active 最低 22問）の範囲内で配分が自動更新される。
- **AC-003**: 精度低下等の異常を検知した場合、自動的に 24:6 へ戻る。
- **AC-004**: 自動適用の証跡が `study_sessions` に正確に保存される。
- **AC-005**: 既存の P7 手動適用予約が共存し、手動予約時は自動適用より優先される。
- **AC-006**: Build PASS / Schema v24 維持。
